Rebuilding a Failed Availability Group

Drop and recreate the failed AG test_new_avg from the healthy primary, keeping the same name, listener, and IP so applications stay unchanged.

AG: test_new_avg Root cause: Failed AG cluster resource (DNS 1257) Primary: Healthy Data-loss risk: None Paths: SSMS or PowerShell

Will the name and IP change?

AG name
Keep it

Dropping the AG frees the name, so you can recreate it as test_new_avg. No change needed.

Listener name
Keep it

Reuse the same listener (network) name. This is what applications resolve and connect to, so keeping it means connection strings don't change.

Listener IP
Stays the same — if you set it

The IP changes only if you assign a different one. Give the recreated listener the same static IP it had before and it stays identical. Use a static IP (not DHCP), and delete the stale listener A record first so the CNO can re-register the same name → IP cleanly in DNS.

Before you start

Fix the root cause first

Grant the cluster name object (CNO) permission on the secure DNS zone — or delete and recreate the stale listener A record. If the 1257 DNS-registration problem isn't fixed, the rebuilt listener fails the exact same way and the new AG resource won't come online.

Also confirm before touching anything:

Pre-flight

Primary is healthy and holds the current data (confirmed). Fresh full backups taken. On both replicas: the [NT AUTHORITY\SYSTEM] login exists with ALTER ANY AVAILABILITY GROUP, CONNECT SQL, and VIEW SERVER STATE, and the mirroring endpoints are STARTED.

Rebuild procedure

No SSMS? Run it all from PowerShell

Every T-SQL block below can be run through Invoke-Sqlcmd — install the module once with Install-Module SqlServer, then wrap any query. Step 7 (the only SSMS-wizard step) has a full native-PowerShell version.

PowerShell · pattern
Import-Module SqlServer
# wrap any T-SQL step like this
Invoke-Sqlcmd -ServerInstance "SQLNODE1" -Database master `
  -Query "DROP AVAILABILITY GROUP [test_new_avg];"

  1. Fix the CNO / DNS permission

    If you haven't already, grant the CNO rights on the secure DNS zone and remove the stale listener A record. This is what lets the recreated listener register to the same name and IP.

  2. Back up the primary databases

    The rebuild is safe for primary data, but take fresh full backups first as a safety net.

    SQL · primary
    BACKUP DATABASE [YourDB]
TO DISK = N'D:\Backup\YourDB_prerebuild.bak'
WITH COMPRESSION, CHECKSUM, INIT;

  3. Drop the availability group

    Run on the primary. The primary's databases stay online as standalone read/write databases, and the failed WSFC cluster resource is removed automatically.

    SQL · primary
    DROP AVAILABILITY GROUP [test_new_avg];

  4. Verify the cluster resource is gone

    Confirm no orphaned resource remains. If one lingers, remove it manually.

    PowerShell
    # should return nothing
Get-ClusterResource | ? Name -eq "test_new_avg"

# only if it lingers as an orphan
Get-ClusterResource "test_new_avg" | Remove-ClusterResource

  5. Drop the stale secondary copies

    The old databases on the secondary are stuck in RESTORING and unusable. Drop them so automatic seeding can recreate them fresh from the primary.

    SQL · secondary
    -- run on the SECONDARY, per database
DROP DATABASE [YourDB];

  6. Confirm prerequisites on both nodes

    Verify the NT AUTHORITY\SYSTEM login and its three grants exist, and that the mirroring endpoints are STARTED — these are the usual reasons a freshly created AG fails to come online.

    SQL · both nodes
    SELECT name, state_desc FROM sys.database_mirroring_endpoints;
-- state_desc must read STARTED

  7. Recreate the AG — same name, listener, and IP

    Option A · SSMS

    Use the New Availability Group Wizard. Reuse the AG name test_new_avg, the same listener name, and the same static IP. Choose Automatic Seeding so the databases re-sync from the primary.

    Option B · PowerShell (if SSMS won't run)

    Edit the variables at the top to match your environment, then run on a machine with the SqlServer module. Endpoints survive the drop, so they're reused automatically.

    PowerShell
    Import-Module SqlServer

# --- adjust to your environment ---
$ag         = "test_new_avg"
$primary    = "SQLNODE1"                 # use SQLNODE1\INSTANCE if named
$secondary  = "SQLNODE2"
$domain     = "yourdomain.com"
$port       = 5022
$dbs        = "YourDB1","YourDB2"
$listener   = "test_new_listener"        # SAME listener name as before
$listenerIp = "10.0.0.50/255.255.255.0"  # SAME static IP / subnet mask
$sqlVersion = 16                         # 16=2022, 15=2019, 14=2017, 13=2016

$primaryPath   = "SQLSERVER:\SQL\$primary\DEFAULT"
$secondaryPath = "SQLSERVER:\SQL\$secondary\DEFAULT"

# --- replica templates (automatic seeding) ---
$rep1 = New-SqlAvailabilityReplica -Name $primary `
  -EndpointUrl "TCP://$($primary).$($domain):$port" `
  -AvailabilityMode SynchronousCommit -FailoverMode Automatic `
  -SeedingMode Automatic -AsTemplate -Version $sqlVersion

$rep2 = New-SqlAvailabilityReplica -Name $secondary `
  -EndpointUrl "TCP://$($secondary).$($domain):$port" `
  -AvailabilityMode SynchronousCommit -FailoverMode Automatic `
  -SeedingMode Automatic -AsTemplate -Version $sqlVersion

# --- create the AG on the primary ---
New-SqlAvailabilityGroup -Name $ag -Path $primaryPath `
  -AvailabilityReplica @($rep1,$rep2) -Database $dbs

# --- join the secondary + allow auto-seeding to create the DBs ---
Join-SqlAvailabilityGroup -Path $secondaryPath -Name $ag
Invoke-Sqlcmd -ServerInstance $secondary `
  -Query "ALTER AVAILABILITY GROUP [$ag] GRANT CREATE ANY DATABASE;"

# --- listener: SAME name + SAME static IP ---
New-SqlAvailabilityGroupListener -Name $listener -StaticIp $listenerIp `
  -Path "$primaryPath\AvailabilityGroups\$ag"

  8. Verify health

    Run on the primary. Both replicas showing CONNECTED and HEALTHY means the rebuild is complete.

    SQL · primary
    SELECT ar.replica_server_name, ars.role_desc,
       ars.connected_state_desc, ars.synchronization_health_desc
FROM sys.dm_hadr_availability_replica_states ars
JOIN sys.availability_replicas ar
  ON ars.replica_id = ar.replica_id;

On seeding size

Automatic seeding pushes the full databases over the network and needs the bandwidth and disk headroom. For very large databases a manual backup/restore seed from the primary can be faster; for a test-sized AG, automatic seeding is simplest.
